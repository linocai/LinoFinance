### Backend API contract

`GET /api/v1/cash-flow-items`

| Param | Type | Default | Notes |
|---|---|---|---|
| `status` | string | none | unchanged |
| `date_from` | date | none | unchanged |
| `date_to` | date | none | unchanged |
| `include_cancelled` | bool | `false` | **NEW**. When `false`, server filters out `status="cancelled"` rows. Explicit `status=cancelled` still returns cancelled rows (i.e. an explicit `status` filter overrides the default-hide). |

Behaviour for `include_cancelled`:

```python
if status is not None:
    # explicit status filter wins; include_cancelled ignored.
    statement = statement.where(CashFlowItem.status == status)
elif not include_cancelled:
    statement = statement.where(CashFlowItem.status != "cancelled")
```

`POST /api/v1/cash-flow-items/{id}/cancel`

| Before state | After state | HTTP | Body |
|---|---|---|---|
| `expected` | `cancelled` | 200 | `CashFlowItemRead` (cancelled) |
| `confirmed` | `cancelled` | 200 | `CashFlowItemRead` (cancelled) |
| `cancelled` (already) | `cancelled` (no-op) | **200** (was 400) | `CashFlowItemRead` (cancelled) |
| `settled` | `settled` | **400** (unchanged) | `{"detail": "Settled cash flow items cannot be cancelled"}` |

Implementation in `set_cash_flow_status`:

```python
def set_cash_flow_status(db, item_id, status, commit=True):
    item = _get_cash_flow_or_raise(db, item_id)
    if status not in {"expected", "confirmed", "cancelled"}:
        raise LedgerValidationError("Unsupported cash flow status")
    if item.status == status:
        # idempotent no-op for any matching status
        return CashFlowItemRead.model_validate(item)
    if item.status == "settled":
        raise LedgerValidationError("Settled cash flow items cannot be changed")
    if item.status == "cancelled":
        # cancelled is terminal for non-cancel targets
        raise LedgerValidationError("Cancelled cash flow items cannot be changed")
    if status == "confirmed" and item.status != "expected":
        raise LedgerValidationError("Only expected cash flow items can be confirmed")
    item.status = status
    if commit:
        db.commit()
        return get_cash_flow_item(db, item_id)
    db.flush()
    return CashFlowItemRead.model_validate(item)
```

The key shift: the `item.status == status` early-return is unconditional, so
re-cancel is a no-op regardless of whether the item is already cancelled.
Settled-to-anything still raises; cancelled-to-anything-else still raises.

**File**: `frontend/LinoFinance/Features/Entries/EntriesView.swift:136` + the

try? await environment.reimbursementsViewModel.refresh()
try? await environment.creditViewModel.refresh()
try? await environment.reportsViewModel.refresh()

silent-swallow pattern flagged in M5.

### M5 — `try? await refresh()` swallows errors

**File**: `frontend/LinoFinance/Features/Shared/ConnectedViewModels.swift:23`
and **every** other `try? await ...Refresh()` call inside any view's
`.task { ... }` block.

**Rule for v1.1.5**:

- **inside view `.task { ... }` blocks** at first appearance, `try?` is OK
  (user expects silent first-load attempts; the error path lands in the
  ViewModel's `errorMessage` already via the `catch` block in `refresh()`).
- **inside user-initiated actions** (button taps, confirm closures, etc.),
  use `try await` and surface `error.localizedDescription` to
  `environment.lastErrorMessage` (existing pattern in EntriesView).
- **after a write operation** (`create`, `confirm`, `cancel`, `settle`,
  `void`, …), the refresh must be `try await` so a stale UI never silently
  outlives a failure to refetch.

Search and replace audit. Builder runs:

```bash
grep -rn "try? await .*\.refresh\(\)" frontend/LinoFinance/Features
```

For each hit, decide using the rule above. There are roughly 20–30 hits;
the majority are inside `.task { ... }` and stay as-is. The ones to flip
are inside post-write closures in `EntriesView`, `CashFlowView`,
`ReimbursementsView`, `CreditView`, `AIWorkspaceView`. Builder must commit
a checklist showing which lines were flipped vs intentionally kept.

**Acceptance**: provoke a backend 500 on confirm (e.g. by sending an
invalid payload); the user sees the error banner, not a stale list.

## P6 — macOS Release Build + `.app` Swap + Deploy Script (ops)

### Goal

Produce a runnable `LinoF.app` at `/Users/linotsai/Applications/LinoF.app`
with `CFBundleShortVersionString` and `CFBundleVersion` both `1.1.5`, and
ship a one-command production deploy script.

### Steps

1. From `release/v1.1.5` clean, verify all version strings:

   ```bash
   grep -E "1\.1\.2" backend/pyproject.toml backend/app/core/config.py | wc -l   # expect ≥ 2
   grep -c "1\.1\.2" frontend/LinoFinance.xcodeproj/project.pbxproj              # expect 12
   ```

2. macOS Release build:

   ```bash
   xcodebuild \
     -project frontend/LinoFinance.xcodeproj \
     -scheme LinoFinance \
     -configuration Release \
     -derivedDataPath frontend/.derivedData \
     -destination 'platform=macOS' \
     build
   ```

3. Confirm the produced app:

   ```bash
   APP=frontend/.derivedData/Build/Products/Release/LinoF.app
   /usr/libexec/PlistBuddy -c "Print :CFBundleShortVersionString" "$APP/Contents/Info.plist"
   /usr/libexec/PlistBuddy -c "Print :CFBundleVersion" "$APP/Contents/Info.plist"
   ```

   Both must print `1.1.5`. If not, P0 missed a `MARKETING_VERSION` or
   `CURRENT_PROJECT_VERSION` entry — fix and rebuild.

4. Back up the existing installed app, then swap:

   ```bash
   DEST=/Users/linotsai/Applications/LinoF.app
   if [ -d "$DEST" ]; then
       mv "$DEST" "/Users/linotsai/Applications/LinoF.app.bak-$(date -u +%Y%m%dT%H%M%SZ)"
   fi
   cp -R "$APP" "$DEST"
   ```

   No codesign / notarization step. Self-use only.

5. Launch sanity:

   ```bash
   open /Users/linotsai/Applications/LinoF.app
   ```

   App should reach the Dashboard (or the Settings connection screen if no
   token) without hanging.

6. Write `scripts/deploy-api.sh` (new file, mode `0755`). Behaviour:

   - SSH to `deploy@118.178.122.194` (use `~/.ssh/config` host alias `hz`),
   - generate UTC release stamp `YYYYMMDD-HHMMSS`,
   - `rsync` `backend/` (excluding `.venv`, `__pycache__`, `.local`,
     `.backups`, `tests`) to `/opt/linofinance/app/releases/<stamp>/backend/`,
   - on remote, in that release dir:
     `python3 -m venv .venv && .venv/bin/pip install -e . --quiet`,
   - on remote, run `python scripts/production_migrate.py` (this already
     takes a pre-migration backup as per `docs/deployment.md`),
   - flip the symlink `/opt/linofinance/app/current` atomically (write to
     `current.new`, `mv -T current.new current`),
   - `sudo systemctl restart linofinance-api`,
   - `curl -fsS https://lf.linotsai.top/api/v1/health` and parse the JSON;
     fail the script if `version != "1.1.5"` or `status != "ok"`,
   - on failure, the script must:
     (a) print the failing curl body,
     (b) print the rollback command `ln -sfn /opt/linofinance/app/releases/<prev-stamp> /opt/linofinance/app/current && sudo systemctl restart linofinance-api`,
     (c) exit non-zero.

   Script must be **idempotent if rerun with the same stamp** — i.e. it
   refuses to overwrite an existing release dir, but cheerfully rebuilds
   the venv if rerun explicitly with `FORCE=1`.

   The script does **not** push commits, does **not** create tags, does
   **not** alter the local repo. Only deploy.

7. Dry-run the script with `--dry-run` (it should print every command
   without executing). The dry-run output is the verification artifact
   that goes into the changelog.

### Acceptance

- `LinoF.app` opens, shows `设置 → 关于` (or the existing version surface)
  displaying `1.1.5`.
- `scripts/deploy-api.sh --dry-run` prints the full plan without
  contacting hz.
- `scripts/deploy-api.sh` (live run) is **not** required for the patch tag
  — user runs it on their own time. The plan only requires that the
  script exists and dry-runs clean.

### Rollback

- For the `.app`: rename the `.bak-*` directory back over `LinoF.app`.
- For the deploy script: if a live deploy fails, the script prints the
  symlink rollback command; user runs it manually. The remote
  `production_migrate.py` already wrote a pre-migration backup before any
  schema change, so DB rollback uses `restore_postgres.py` with
  `--confirm RESTORE_LINOFINANCE`.

## 4. API Contract Diff vs v1.1.0

Only one endpoint changes shape; everything else is byte-compatible.

### Modified — `GET /api/v1/cash-flow-items`

```text
GET /api/v1/cash-flow-items?[status=]&[date_from=]&[date_to=]&[include_cancelled=false]
Response 200: CashFlowItemRead[]
```

Old clients (no `include_cancelled`) see fewer rows — cancelled rows are
silently hidden. This is an intentional, user-visible default change.

### Modified — `POST /api/v1/cash-flow-items/{id}/cancel`

Status table redux:

| Before | After | Status |
|---|---|---|
| any non-terminal | cancelled | 200 |
| cancelled | cancelled | **200 (was 400)** |
| settled | settled | 400 |

Error code for settled is now:

```json
{"detail": "Settled cash flow items cannot be changed"}
```

(was: `"Settled or cancelled cash flow items cannot be changed"`).
Frontend does **not** parse this string; the change is safe.

### Unchanged

Every other endpoint, including `/health`, returns the same shape. Only
`version` field of `/health` changes from `1.1.0` to `1.1.5`.

### Auth

Unchanged: `Authorization: Bearer <LINOFINANCE_API_AUTH_TOKEN>` for any
non-public route. Token requirement, rate-limit headers, request-ID
headers all unchanged.
