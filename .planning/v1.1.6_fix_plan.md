## 0. Project Goal

Ship `v1.1.6` as a focused finance-UX patch that:

1. introduces a third account type `investment` (alongside `balance` and
   `credit`), without an Alembic migration (the `accounts.type` column is
   already a free `String(32)`),
2. backfills existing `funds` / `stock` accounts on production to
   `type='investment'` while keeping their names intact (after a
   builder-run name audit on production data — see P1),
3. rewrites `GET /dashboard/summary` to return per-currency disposable,
   per-currency investment total, per-currency today-pnl, per-currency
   30-day cash-flow net, and a net-worth that **includes investment
   accounts**,
4. adds a `POST /accounts/{id}/daily-pnl` endpoint that records a daily
   P&L delta against an `investment` account by submitting the **new
   current balance**, with the server computing the delta and writing one
   `account_adjustments` row plus one `audit_logs` row,
5. reorders the macOS dashboard KPI grid and iOS hero card to the new
   user-decided layout (未来一月可支配 / 投资账户 / 净资产 / 未来 30 天
   现金流),
6. groups the account list into three sections (余额 / 投资 / 信用) on
   both macOS and iOS, and surfaces a daily-pnl quick-input affordance:
   - macOS: a compact form embedded into the existing macOS sidebar,
   - iOS: a `+` button on each investment-account row that presents a
     sheet,
7. adds "投资账户" to the New Account / Edit Account type pickers,
8. bumps every visible version string to `1.1.6` (backend pyproject,
   `config.app_version`, all 12 Xcode `MARKETING_VERSION` /
   `CURRENT_PROJECT_VERSION` entries),
9. produces a working macOS `.app` at
   `/Users/linotsai/Applications/LinoF.app` with
   `CFBundleShortVersionString = CFBundleVersion = 1.1.6`.

Out of scope: AI module reform, new Alembic migrations (the patch is
schema-stable — `accounts.type` is free string, `account_adjustments`
already has every column we need), production live deploy (handled by
user manually after sign-off), iOS device install (still blocked on
Apple provisioning).

### Dashboard summary — new response shape

Endpoint: `GET /api/v1/dashboard/summary`

**Existing fields (kept verbatim, do NOT remove):**

```json
{
  "base_currency": "CNY",
  "balance_total_cny": "12345.67",
  "credit_liability_total_cny": "2333.00",
  "net_worth_cny": "10012.67",
  "draft_entry_count": 0,
  "confirmed_entry_count": 42,
  "voided_entry_count": 1
}
```

`balance_total_cny` continues to mean **non-credit, non-investment
balance accounts only**. `credit_liability_total_cny` continues to mean
credit accounts. **`net_worth_cny` semantics CHANGE**:

```
net_worth_cny = balance_total_cny + investment_total_cny - credit_liability_total_cny
```

This is intentionally a backwards-incompatible change to the numeric
value of `net_worth_cny`. Old clients that read this field still get a
valid number; they just see "net worth has gone up" once a real
investment account exists. This is exactly what the user wants.

**New fields (additive):**

```json
{
  "investment_total_cny": "5000.00",
  "investment_total_by_currency": [
    {"currency": "CNY", "amount": "3000.00"},
    {"currency": "USD", "amount": "280.00"}
  ],
  "today_pnl_by_currency": [
    {"currency": "CNY", "amount": "-120.50"},
    {"currency": "USD", "amount": "5.00"}
  ],
  "disposable_30d_by_currency": [
    {"currency": "CNY", "amount": "8000.00"},
    {"currency": "USD", "amount": "0.00"}
  ],
  "cash_flow_30d_by_currency": [
    {"currency": "CNY", "amount": "-4000.00"},
    {"currency": "USD", "amount": "0.00"}
  ]
}
```

Field semantics:

| Field | Definition |
|---|---|
| `investment_total_cny` | Σ of `current_balance` of all `active`, `include_in_net_worth=true`, `type='investment'` accounts, converted to CNY using the latest manual rate on `date.today()`. Decimal, quantized 2dp. |
| `investment_total_by_currency` | Same as above but grouped **without** CNY conversion. One row per distinct currency that has at least one investment account. Hidden in JSON when the array is empty (i.e. no investment accounts exist). |
| `today_pnl_by_currency` | Σ of `account_adjustments.delta_amount` where the linked account has `type='investment'`, `account_adjustments.source = 'investment_daily'`, and `account_adjustments.created_at::date = today_utc()`. Group by `account_adjustments.currency`. Decimal, may be negative. Rows with `amount == 0` still included if at least one daily-pnl adjustment exists for that currency today (so the user sees "0 today" rather than nothing). If **no** daily-pnl rows exist today, the array is `[]`. |
| `disposable_30d_by_currency` | "未来一月可支配" — for each currency in `{CNY, USD}` separately: Σ (`current_balance` of all `active`, `include_in_net_worth=true`, **non-investment**, non-credit accounts in that currency) **plus** the net cash flow in that currency over today..today+30. **Excludes investment accounts**, **excludes credit accounts**. Output: one row per currency with nonzero data (rule: include CNY always; include USD only if either side is non-zero — i.e. there is at least one USD balance account or at least one USD cash flow in the next 30 days). |
| `cash_flow_30d_by_currency` | Σ of `(direction inflow ? +amount : -amount)` for `cash_flow_items` with `status in ('expected', 'confirmed')` and `expected_date in [today, today+30)`, grouped by the cash-flow item's own `currency`. CNY always present; USD present only if non-zero. |

Decimal rendering rule: use the existing `format_decimal` serializer
from `backend/app/schemas/entry.py` for every Decimal field above
(matches v1.1.0+ convention).

**Pydantic schema** (additive, do **not** remove existing fields):

```python
# backend/app/schemas/dashboard.py
class CurrencyAmount(BaseModel):
    currency: str
    amount: Decimal

    @field_serializer("amount")
    def serialize_amount(self, value: Decimal) -> str:
        return format_decimal(value)


class DashboardSummary(BaseModel):
    # existing — keep
    base_currency: str
    balance_total_cny: Decimal
    credit_liability_total_cny: Decimal
    net_worth_cny: Decimal
    draft_entry_count: int
    confirmed_entry_count: int
    voided_entry_count: int

    # new in v1.1.6
    investment_total_cny: Decimal = Decimal("0")
    investment_total_by_currency: list[CurrencyAmount] = []
    today_pnl_by_currency: list[CurrencyAmount] = []
    disposable_30d_by_currency: list[CurrencyAmount] = []
    cash_flow_30d_by_currency: list[CurrencyAmount] = []

    @field_serializer("balance_total_cny", "credit_liability_total_cny",
                      "net_worth_cny", "investment_total_cny")
    def serialize_decimal(self, value: Decimal) -> str:
        return format_decimal(value)
```

**Service implementation sketch**
(`backend/app/services/dashboard.py`):

```python
def get_dashboard_summary(db: Session) -> DashboardSummary:
    today = date.today()
    balance_total_cny = Decimal("0")
    credit_liability_total_cny = Decimal("0")
    investment_total_cny = Decimal("0")

    by_ccy_balance: dict[str, Decimal] = {}      # non-credit non-investment
    by_ccy_invest: dict[str, Decimal] = {}

    accounts = db.execute(
        select(Account).where(
            Account.status == "active",
            Account.include_in_net_worth.is_(True),
        )
    ).scalars()
    for account in accounts:
        if account.type == "credit":
            converted, _ = convert_to_cny(db, account.current_liability, account.currency, today)
            credit_liability_total_cny += converted
        elif account.type == "investment":
            converted, _ = convert_to_cny(db, account.current_balance, account.currency, today)
            investment_total_cny += converted
            by_ccy_invest[account.currency] = by_ccy_invest.get(account.currency, Decimal("0")) + account.current_balance
        else:  # balance + anything new
            converted, _ = convert_to_cny(db, account.current_balance, account.currency, today)
            balance_total_cny += converted
            by_ccy_balance[account.currency] = by_ccy_balance.get(account.currency, Decimal("0")) + account.current_balance

    today_pnl = _today_pnl_by_currency(db, today)
    cash_flow_30d = _cash_flow_30d_by_currency(db, today)
    disposable = _disposable_30d(by_ccy_balance, cash_flow_30d)

    entry_counts = _entry_counts_by_status(db)

    return DashboardSummary(
        base_currency="CNY",
        balance_total_cny=quantize_money(balance_total_cny),
        credit_liability_total_cny=quantize_money(credit_liability_total_cny),
        net_worth_cny=quantize_money(balance_total_cny + investment_total_cny - credit_liability_total_cny),
        draft_entry_count=entry_counts.get("draft", 0),
        confirmed_entry_count=entry_counts.get("confirmed", 0),
        voided_entry_count=entry_counts.get("voided", 0),
        investment_total_cny=quantize_money(investment_total_cny),
        investment_total_by_currency=_pack(by_ccy_invest),
        today_pnl_by_currency=today_pnl,
        disposable_30d_by_currency=disposable,
        cash_flow_30d_by_currency=cash_flow_30d,
    )
```

The helpers `_today_pnl_by_currency`, `_cash_flow_30d_by_currency`,
`_disposable_30d`, `_pack` go in the same file. Use the existing
`from datetime import timedelta`, `select(CashFlowItem)` patterns from
`backend/app/services/report.py:138-197`. The window is **`expected_date
in [today, today + 30)`**, identical to `daily_net_window(days=30)`.

`_pack` rule for `investment_total_by_currency`: include the entry only
if `amount != 0`. For `today_pnl_by_currency`: include a row if there
exists at least one matching adjustment today, even with `amount == 0`.
For `disposable_30d_by_currency` and `cash_flow_30d_by_currency`:
always include CNY; include USD only if either contributing side is
non-zero.

### Daily P&L endpoint

`POST /api/v1/accounts/{account_id}/daily-pnl`

Request body:

```json
{
  "new_balance": "12340.50",
  "as_of_date": "2026-05-24",     // optional, defaults to today (server time)
  "note": "盘后调整"               // optional
}
```

Response `201`:

```json
{
  "adjustment_id": "<uuid>",
  "account_id": "<uuid>",
  "currency": "CNY",
  "balance_before": "12500.00",
  "balance_after": "12340.50",
  "delta_amount": "-159.50",
  "as_of_date": "2026-05-24",
  "source": "investment_daily"
}
```

Validation rules:

| Rule | Error |
|---|---|
| Account must exist | `404 {"detail":"Account not found"}` |
| Account `type` must be `'investment'` | `400 {"detail":"Daily P&L is only allowed on investment accounts"}` |
| Account `status` must be `'active'` | `400 {"detail":"Daily P&L is not allowed on inactive accounts"}` |
| `new_balance` must parse as Decimal | `422` (pydantic) |
| `new_balance` may be negative (margin / short positions). Allowed. | — |
| `as_of_date` may be today or any past date; future dates rejected | `400 {"detail":"Daily P&L date cannot be in the future"}` |
| Delta `new_balance - current_balance` of `0` is allowed (records a "no change" data point); does NOT raise. | — |

Side effects on success (single DB transaction):

1. INSERT one `account_adjustments` row:
   - `account_id` = path param,
   - `reason` = `"daily_pnl"`,
   - `delta_amount` = `new_balance - account.current_balance`,
   - `currency` = `account.currency`,
   - `balance_before` = `account.current_balance` (before update),
   - `balance_after` = `new_balance`,
   - `source` = `"investment_daily"`,
   - `note` = `payload.note`,
   - `created_by` = `"user"` (no per-user yet).
2. UPDATE `accounts.current_balance` = `new_balance` for this account.
3. INSERT one `audit_logs` row with
   `action_type="account.daily_pnl"`,
   `target_type="account"`, `target_id=account.id`,
   `before_snapshot={"current_balance": "<before>"}`,
   `after_snapshot={"current_balance":"<after>","delta":"<delta>","as_of_date":"<iso>"}`.

The endpoint does **not** read or write any cash flow item or entry —
investment P&L is intentionally **not** an "entry" in the ledger. It is
purely a balance reconciliation event, hence the reuse of
`account_adjustments` with the new `source` value.

Old clients that read only the original 7 fields still work. They will

### v1.1.6 backfill SQL 草案（占位，由 builder 在 P1 填实际 UUID）

```sql
-- Step A — read-only audit (builder runs on hz first, shows user)
SELECT id, name, type, currency, current_balance
  FROM accounts
 WHERE status = 'active'
   AND type = 'balance'
   AND (
         lower(name) LIKE '%fund%'
      OR lower(name) LIKE '%stock%'
      OR lower(name) LIKE '%基金%'
      OR lower(name) LIKE '%股票%'
      OR lower(name) LIKE '%证券%'
      OR lower(name) LIKE '%理财%'
      OR lower(name) LIKE '%投资%'
   )
 ORDER BY name;

-- Step B — after user confirms specific UUIDs, run as one transaction
BEGIN;
UPDATE accounts SET type = 'investment'
 WHERE id IN ('<uuid1>', '<uuid2>', ... );

INSERT INTO audit_logs (id, actor, action_type, target_type, target_id,
                         before_snapshot, after_snapshot, note, created_at)
SELECT gen_random_uuid()::text,
       'admin:v1.1.6-backfill',
       'account.type.update',
       'account',
       a.id,
       jsonb_build_object('type','balance'),
       jsonb_build_object('type','investment'),
       'v1.1.6 backfill: classify investment accounts',
       now()
  FROM accounts a
 WHERE a.id IN ('<uuid1>', '<uuid2>', ... );
COMMIT;
```

The user takes a `backend/scripts/backup_postgres.py` snapshot named
something like `linofinance-before-v1.1.6-backfill-<UTC>.dump` before
running Step B. Audit row count after `COMMIT` must equal the UPDATE
row count.
