> Patch release `v1.1.7` on top of the deployed `v1.1.6` release.

## 0. Project Goal

Ship `v1.1.7` as a focused cash-flow UX patch that:

1. introduces a single new backend endpoint `PATCH
   /api/v1/cash-flow-items/{id}` allowing every editable field of a cash
   flow item to be updated **while** the item is still in
   `status in ('expected', 'confirmed')` тАФ `settled` and `cancelled` rows
   stay locked,
2. enforces currency consistency on edit: when `currency != 'CNY'` the
   caller must also send an `exchange_rate_id` (same rule as the

   sheet,

   /{id}/confirm` and `POST /{id}/cancel` keep working; the old "confirm

   non-LinoFinance integration calls it; only the LinoFinance UI stops

### `CashFlowItemUpdate` schema

```python
# backend/app/schemas/cash_flow.py
from pydantic import BaseModel, Field
from typing import Optional


class CashFlowItemUpdate(BaseModel):
    title: Optional[str] = Field(default=None, min_length=1, max_length=200)
    direction: Optional[str] = Field(default=None, pattern="^(inflow|outflow|transfer)$")
    cash_flow_type: Optional[str] = Field(
        default=None,
        pattern="^(salary|rent_income|reimbursement|subscription|credit_repayment|installment|one_time|other)$",
    )
    amount: Optional[Decimal] = Field(default=None, gt=0)
    currency: Optional[str] = Field(default=None, min_length=3, max_length=3)
    exchange_rate_id: Optional[str] = None
    converted_cny_amount: Optional[Decimal] = None
    expected_date: Optional[DateType] = None
    account_id: Optional[str] = None      # explicit None means "unlink"
    category_id: Optional[str] = None     # explicit None means "unlink"
    recurrence_rule: Optional[str] = Field(default=None, max_length=200)
    note: Optional[str] = None
```

**Critical semantics for account/category nulling**: Pydantic cannot
distinguish "field absent" from "field present and null" with this
shape alone. To allow the UI to clear `account_id` / `category_id`,
adopt the **sentinel-via-model_fields_set** rule:

```python
provided_fields = payload.model_fields_set  # set of field names actually present in the JSON
```

A field is "intentionally provided" iff its name appears in
`provided_fields`. If `"account_id" in provided_fields` and the value
is `None`, that means "unlink". If the key is absent, leave the
existing `account_id` untouched. Same rule for `category_id`,
`exchange_rate_id`, `converted_cny_amount`, `recurrence_rule`, `note`.

Status, `linked_*_id`, and `linked_entry_id` are **NOT** in
`CashFlowItemUpdate`. Those go through their existing endpoints
(`/confirm`, `/cancel`, `/settle`) so v1.1.7 PATCH cannot accidentally
sidestep settlement.

### Service implementation

```python
# backend/app/services/cash_flow.py
def update_cash_flow_item(
    db: Session,
    item_id: str,
    payload: CashFlowItemUpdate,
) -> CashFlowItemRead:
    item = _get_cash_flow_or_raise(db, item_id)
    if item.status in {"settled", "cancelled"}:
        raise LedgerValidationError(
            "Settled or cancelled cash flow items cannot be edited"
        )

    provided = payload.model_fields_set

    # Apply scalar overrides where present.
    if "title" in provided:
        item.title = payload.title
    if "direction" in provided:
        item.direction = payload.direction
    if "cash_flow_type" in provided:
        item.cash_flow_type = payload.cash_flow_type
    if "expected_date" in provided:
        item.expected_date = payload.expected_date
    if "account_id" in provided:
        item.account_id = payload.account_id
    if "category_id" in provided:
        item.category_id = payload.category_id
    if "recurrence_rule" in provided:
        item.recurrence_rule = payload.recurrence_rule
    if "note" in provided:
        item.note = payload.note

    # Money/currency triplet: amount + currency + exchange_rate_id +
    # converted_cny_amount are interlocked. If ANY of the four is
    # provided, recompute the trio off the (new or existing) values.
    money_provided = bool(
        provided & {"amount", "currency", "exchange_rate_id", "converted_cny_amount"}
    )
    if money_provided:
        new_amount = quantize_money(payload.amount) if "amount" in provided else item.amount
        new_currency = (payload.currency or item.currency).upper() if "currency" in provided else item.currency
        new_rate_id = payload.exchange_rate_id if "exchange_rate_id" in provided else item.exchange_rate_id
        new_converted = payload.converted_cny_amount if "converted_cny_amount" in provided else None

        # Currency-consistency rule: non-CNY MUST come with exchange_rate_id.
        if new_currency != "CNY" and not new_rate_id:
            raise LedgerValidationError(
                "exchange_rate_id is required when currency is not CNY"
            )

        converted, resolved_rate_id = _resolve_payload_conversion(
            db, new_amount, new_currency, item.expected_date,
            new_rate_id, new_converted,
        )
        item.amount = new_amount
        item.currency = new_currency
        item.exchange_rate_id = resolved_rate_id
        item.converted_cny_amount = converted

    # Re-validate optional links against the *new* state.
    _validate_update_links(db, item)

    db.flush()
    db.commit()
    return get_cash_flow_item(db, item_id)


def _validate_update_links(db: Session, item: CashFlowItem) -> None:
    """Like _validate_optional_links but read from the model post-mutation."""
    if item.account_id is not None:
        account = db.get(Account, item.account_id)
        if account is None:
            raise LedgerValidationError("Account not found")
        if account.currency != item.currency:
            raise LedgerValidationError(
                "Cash flow currency must match linked account currency"
            )

    if item.category_id is not None and db.get(Category, item.category_id) is None:
        raise LedgerValidationError("Category not found")
```

`_resolve_payload_conversion` already exists тАФ the new code calls it
with the post-mutation currency/amount/expected_date trio so the
exchange-rate lookup uses fresh inputs.

**Salary recurrence note**: PATCHing `recurrence_rule` only changes the
**this row's** rule string. The plan does **not** trigger
re-materialisation of a 12-month salary series тАФ that would touch
multiple rows and is out of scope. If the user wants to retime an
entire salary series, they delete and recreate (existing behaviour).

### Request DTO

```swift
// APIRequests.swift
struct CashFlowItemUpdateRequest: Encodable {
    var title: String?
    var direction: String?
    var cashFlowType: String?
    var amount: DecimalValue?
    var currency: CurrencyCode?
    var exchangeRateId: String?
    var convertedCnyAmount: DecimalValue?
    var expectedDate: Date?
    // Use a double-Optional sentinel so the encoder distinguishes
    // "absent" (.none) from "explicit null" (.some(.none)).
    // Swift's default encoder elides nil, which would prevent unlinking;
    // wrap the two link fields in `Nullable<String>` instead.
    var accountId: Nullable<String>?
    var categoryId: Nullable<String>?
    var recurrenceRule: String?
    var note: String?
}

/// Wrapper that lets a JSON encoder emit `null` for an explicit clear
/// while still allowing "absent" via the outer Optional.
enum Nullable<T: Encodable>: Encodable {
    case value(T)
    case null

    func encode(to encoder: Encoder) throws {
        var container = encoder.singleValueContainer()
        switch self {
        case .value(let v): try container.encode(v)
        case .null: try container.encodeNil()
        }
    }
}
```

## 5. API Contract Diff vs v1.1.6

### New тАФ `PATCH /api/v1/cash-flow-items/{id}`

```text
PATCH /api/v1/cash-flow-items/{id}
Body (all keys optional; absent = leave unchanged; explicit null on
account_id / category_id = unlink):
{
  title?: string (1..200),
  direction?: "inflow"|"outflow"|"transfer",
  cash_flow_type?: "salary"|"rent_income"|"reimbursement"|"subscription"
                  |"credit_repayment"|"installment"|"one_time"|"other",
  amount?: decimal (> 0),
  currency?: string (3),
  exchange_rate_id?: string,
  converted_cny_amount?: decimal,
  expected_date?: ISO date,
  account_id?: string | null,
  category_id?: string | null,
  recurrence_rule?: string (<=200),
  note?: string
}

тЖТ 200  CashFlowItemRead (same shape as POST returns; status preserved)
тЖТ 400  see error table below
тЖТ 404  { detail: "Cash flow item not found" }
тЖТ 422  pydantic validation (invalid enum value, malformed decimal, etc.)
```

The endpoint is **idempotent** in the sense that PATCHing the same body
twice yields the same final state, but writes happen on every call
(`updated_at` if any), so callers should not retry blindly without
checking.

### Error codes table

| Endpoint | Code | Detail |
|---|---|---|
| `PATCH /cash-flow-items/{id}` | 404 | `"Cash flow item not found"` |
| `PATCH /cash-flow-items/{id}` | 400 | `"Settled or cancelled cash flow items cannot be edited"` |
| `PATCH /cash-flow-items/{id}` | 400 | `"exchange_rate_id is required when currency is not CNY"` |
| `PATCH /cash-flow-items/{id}` | 400 | `"Account not found"` |
| `PATCH /cash-flow-items/{id}` | 400 | `"Cash flow currency must match linked account currency"` |
| `PATCH /cash-flow-items/{id}` | 400 | `"Category not found"` |
| `PATCH /cash-flow-items/{id}` | 400 | `"converted_cny_amount does not match the exchange rate"` |
| `PATCH /cash-flow-items/{id}` | 422 | pydantic (regex failures, decimal parse, length) |

### Unchanged

`POST /cash-flow-items` (create), `POST /cash-flow-items/{id}/confirm`,
`POST /cash-flow-items/{id}/cancel`, `POST /cash-flow-items/{id}/settle`,
`GET /cash-flow-items`, `GET /cash-flow-items/{id}` тАФ all keep their
v1.1.6 behaviour. The LinoFinance UI stops calling `/confirm` after
v1.1.7, but the route stays for backwards compatibility with any
third-party integration.

`/health` `version` field flips from `1.1.6` to `1.1.7`. Nothing else
in `/health` changes.

### Auth

Unchanged: `Authorization: Bearer <LINOFINANCE_API_AUTH_TOKEN>` for any
non-public route.
