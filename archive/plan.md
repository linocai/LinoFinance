# LinoFinance 施工 Plan

本文档是当前项目施工总入口，综合：

- `LinoFinance前置计划.md`：业务逻辑、后端模型、AI Agent、报表、通知、验收规则。
- `LinoFinance前端设计方向.md`：iOS/macOS 信息架构、交互、组件、页面与前端 MVP 顺序。

## 0. 最新确认决策

| 决策点 | 施工口径 |
|---|---|
| 汇率 | V1 先手动维护，不接自动汇率 API；初始 USD/CNY 汇率为 `6.8`。历史记录保存当时汇率，不因后续汇率变化回写。 |
| 信用卡账单 | 需要独立的账单周期对象 `CreditStatementCycle`，不要只靠账单日/还款日和账户字段硬算。 |
| 报销净支出 | 报表都要支持：报销前支出、预计报销抵扣、已批准抵扣、已到账抵扣、个人净支出。 |
| AI 自动确认阈值 | 默认阈值为 `1000 CNY`。小于等于该阈值的低风险动作可进入自动确认候选，但仍必须满足字段完整、可回滚、可审计。 |
| 数据部署 | 整个 DB 部署到云服务器；iOS/macOS 通过域名访问后端 API。客户端本地存储只作为缓存、离线草稿或临时队列，不作为主数据源。 |
| 数据导出 | V1 只做 CSV 导出。JSON、SQLite 备份、PDF 月报不进入第一版导出范围。 |

## 1. 总体施工原则

1. 后端是账本真相源，前端通过 API 读写；所有金额、账户余额、信用负债、报销净额和报表聚合以服务端计算为准。
2. 所有金额必须保留原始币种，同时保存折算人民币金额和所用汇率；前端展示原币 + CNY 折算。
3. 正式记录必须完整，草稿记录可以不完整；草稿不影响账户余额和正式报表。
4. 信用卡消费计入消费和信用负债，信用卡还款只算账户转移，不重复计入消费。
5. 可报销消费必须同时产生正式支出、报销对象、未来应收现金流。
6. AI 只能通过结构化动作操作账本，所有动作分级、可确认、可回滚、可审计。
7. iOS 优先快速记账和一眼总览；macOS 优先表格、批量编辑、报表、对账和管理。

## 2. Phase 0：项目骨架与工程基线

目标：先把可持续施工的仓库、服务、客户端和约定搭好。

后端：

- 确定后端技术栈、包管理、配置方式和目录结构。
- 建立 API 服务骨架、健康检查、错误响应格式、请求日志。
- 建立数据库迁移机制和云服务器部署预案。
- 建立环境配置：本地开发、测试、生产，数据库连接用环境变量。
- 建立基础鉴权占位：单用户模式可以先简化，但 API 边界要预留鉴权。

前端：

- 建立 iOS/macOS SwiftUI 工程骨架。
- 建立共享模块：Models、Services、Repositories、DesignSystem、Features。
- 建立 API Client 抽象，所有功能通过域名 API 访问后端。
- SwiftData/Core Data 如使用，只作为缓存和离线草稿队列，不作为主存储。
- 建立 DesignSystem v0：`MoneyText`、`StatusTag`、`CurrencyBadge`、颜色语义令牌。

验收：

- 本地可以启动后端 API。
- 客户端可以调用健康检查接口。
- Git 仓库干净，有基础 README 或开发启动说明。
- 数据库迁移可以从空库跑通。

## 3. Phase 1：基础账本 MVP

目标：做出最小可用账本，先保证账户余额、分类、单笔/汇总记录和双币种准确。

后端：

- 模型：`CurrencyRate`、`Account`、`Category`、`FinancialEntry`、`EntryCategoryLine`、`AccountMovement`、`AuditLog`。
- 汇率：种子数据写入 USD/CNY = `6.8`，source = `manual`。
- 记录能力：单笔记录、日汇总、多日汇总、草稿、确认、作废、归档。
- 校验规则：正式记录必须有分类明细和账户变动；分类金额和账户变动可对账；金额必须大于 0。
- 账户余额：通过账户变动计算余额，避免前端自行推导。
- API：账户、分类、记录、草稿确认、余额总览。

前端：

- iOS：5 Tab 雏形，先落总览、记账、资产账户。
- macOS：Sidebar + 列表 + 详情 Inspector 雏形。
- 记账表单：单笔记录、汇总记录、多分类、多账户拆分。
- 总览 v0：净资产、余额账户、信用负债占位、本月支出。
- 所有金额展示统一走 `MoneyText`：原币 + 约 CNY。

验收：

- 可以创建 CNY/USD 账户和分类。
- 可以记一笔美元消费，并按 6.8 汇率展示人民币折算。
- 可以记多日汇总，并精确拆到账户。
- 确认记录后账户余额变化正确；草稿不影响余额和报表。

## 4. Phase 2：信用账户与账单周期

目标：信用卡消费、还款、账单周期和信用负债闭环。

后端：

- 扩展 `Account` 支持信用账户字段：额度、账单日、还款日、最低还款、状态等。
- 新增 `CreditStatementCycle`：
  - `id`
  - `creditAccountId`
  - `cycleStartDate`
  - `cycleEndDate`
  - `statementDate`
  - `dueDate`
  - `currency`
  - `statementAmount`
  - `minimumPayment`
  - `paidAmount`
  - `status`: `open`、`statement_generated`、`partially_paid`、`paid`、`overdue`、`closed`
  - `linkedCashFlowItemId`
  - `note`
- 信用消费：增加支出和信用负债，归入对应账单周期。
- 信用还款：余额账户转出，信用账户负债减少，不计入消费分类。
- 自动生成信用卡还款未来现金流：账单已确认则状态为 confirmed，未确认则 expected。
- API：信用账户总览、账单周期列表、还款记录、账单核对基础接口。

前端：

- iOS 资产 Tab 增加信用中心：信用卡卡片、应还、还款日倒计时。
- macOS 信用中心：账单周期列表、账单详情、还款历史、差异核对入口占位。
- 记账表单支持信用账户消费与信用还款。
- 总览加入信用负债和最近还款提醒。

验收：

- 信用卡消费会增加消费和应还金额。
- 信用卡还款不会重复计入消费。
- 一个账单周期能正确收拢周期内消费、应还、已还、剩余。
- 还款现金流可由账单周期自动生成。

## 5. Phase 3：未来现金流

目标：把未来收入、未来支出、还款、订阅、分期、报销等统一放进现金流系统。

后端：

- 模型：`CashFlowItem`、周期规则对象或 recurrence 字段。
- 支持一次性现金流和周期性现金流。
- 状态：`expected`、`confirmed`、`settled`、`cancelled`、`partial`。
- 支持现金流转正式记录：确认发生后生成 `FinancialEntry` 和 `AccountMovement`。
- API：未来 7/30/90 天视图、时间线、状态切换、确认发生。

前端：

- iOS：现金流 Tab，未来 30 天卡片，快速确认已发生。
- macOS：现金流列表、时间轴、日历视图、批量确认/取消。
- 总览增加未来现金流压力卡。

验收：

- 可以创建一次性和周期性未来现金流。
- 可以把 confirmed 现金流确认成正式账。
- 未来现金流不直接影响正式报表，settled 后才影响真实账本。

## 6. Phase 4：报销系统

目标：可报销消费形成支出、应收、现金流、报表视角完整闭环。

后端：

- 模型：`ReimbursementClaim`、`Attachment`。
- 可报销消费确认后自动生成：
  - 原始正式支出。
  - 报销对象。
  - 未来应收现金流。
- 报销状态：草稿/待提交/已提交/已批准/已到账/拒绝/取消。
- 报销到账后生成收入或抵扣型记录，并关联原消费。
- 报表聚合必须支持：
  - 报销前支出。
  - 预计报销抵扣。
  - 已批准抵扣。
  - 已到账抵扣。
  - 个人净支出。
- API：报销中心、状态变更、附件关联、报销视角报表。

前端：

- iOS：报销中心列表、状态标签、预计到账、原消费链接。
- macOS：报销 Kanban、批量提交、批量标记到账、附件管理。
- 报表切换：报销前 / 预计净支出 / 已批准净支出 / 已到账净支出。

验收：

- 一笔可报销信用卡消费能同时产生支出、报销对象和未来应收。
- 报销到账后个人净支出视角正确抵消。
- 未批准、已批准、已到账在报表中都能按不同视角展示。

## 7. Phase 5：分期与订阅

目标：支持大件分期、周期服务订阅和未来现金流压力。

后端：

- 模型：`InstallmentPlan`、订阅规则对象。
- 分期类型 A：一次性消费，未来分期还；消费归属购买当月，未来现金流按期展开。
- 分期类型 B：每期服务，每期才发生消费；适合订阅。
- 支持提前结清、取消、状态追踪。
- API：分期计划、订阅规则、未来期数生成、订阅统计。

前端：

- 分期计划详情：总金额、每期金额、剩余期数、提前结清。
- 订阅列表：下次扣款日、月度/年度统计、即将扣款提醒。
- 信用卡详情内嵌分期视图。

验收：

- 大件分期不会把每期还款重复算成消费。
- 订阅按实际发生周期生成正式记录或现金流。
- 现金流压力能看到未来分期和订阅。

## 8. Phase 6：AI 财务管家与通知

目标：AI 可以辅助记账、分类、生成现金流、管理报销和提醒，但必须结构化、可控、可回滚。

后端：

- 定义结构化动作协议：`CreateEntry`、`CreateCashFlowItem`、`MarkReimbursable`、`CreateInstallmentPlan`、`RecordCreditRepayment`、`GenerateNotificationRule` 等。
- 定义 `AIAction`、`AIPlan`、`AIActionExecution`、回滚记录和审计日志。
- 权限分级：
  - 低风险：字段完整、金额小于等于 `1000 CNY`、不修改历史、不删除数据，可自动确认候选。
  - 中风险：新增未来现金流、标记报销、修改还款金额、批量确认草稿，需要用户确认。
  - 高风险：删除、作废、修改历史汇率、修改已确认大额记录、批量改历史数据，必须强确认。
- 所有 AI 动作写入 `AuditLog`，包含 before/after。
- 通知规则：信用卡还款、现金流、报销、订阅、异常支出。

前端：

- iOS/macOS AI 一句话录入 Sheet：AI / 表单 / 粘贴三入口。
- AI 对话页显示自然语言解释 + `AIPlanCard`，不直接黑箱改账。
- AI 任务中心：待确认计划、最近执行、可回滚、AI 建议规则、异常提醒。
- 通知：UNUserNotificationCenter、Widget/Live Activity/App Intents 逐步接入。

验收：

- AI 能把一句话转成结构化动作卡。
- 小于等于 `1000 CNY` 的低风险完整记账动作可以自动确认或一键确认。
- 高风险动作必须走计划、确认、执行。
- 每个 AI 执行动作都能在审计日志看到。

## 9. Phase 7：分析报表与 CSV 导出

目标：把账本变成可决策的个人财务控制台。

后端：

- 报表聚合 API：
  - 月度总览。
  - 分类支出。
  - 现金流压力。
  - 信用负债趋势。
  - 报销报告。
  - 订阅报告。
- 所有报表支持币种展示和 CNY 汇总。
- 报销相关报表必须支持 Phase 4 中定义的所有视角。
- CSV 导出：
  - entries.csv
  - entry_category_lines.csv
  - account_movements.csv
  - accounts.csv
  - cash_flow_items.csv
  - reimbursement_claims.csv
  - credit_statement_cycles.csv
  - installment_plans.csv
  - audit_logs.csv

前端：

- macOS 报表中心：图表、表格、视角切换、时间范围、币种切换。
- iOS 总览强化：关键指标、异常提醒、AI 财务简报。
- 导出入口：macOS 优先，iOS 可后置；V1 只导出 CSV。

验收：

- 报表可在报销前、预计抵扣、已批准抵扣、已到账抵扣、个人净支出之间切换。
- CSV 导出可以完整复盘账本核心数据。
- 导出的金额包含原币、汇率、CNY 折算。

## 10. Phase 8：云端部署、域名 API 与端到端稳定化

目标：从本地 MVP 进入可长期使用的云端版本。

后端：

- 云服务器部署 DB 和 API 服务。
- 配置域名、HTTPS、反向代理、日志、备份和恢复。
- 建立生产环境迁移流程和备份策略。
- API 增加限流、鉴权、错误追踪和结构化日志。
- 关键业务加集成测试：账本、信用卡、报销、现金流、AI 动作。

前端：

- API Base URL 切换到域名。
- 登录/鉴权接入真实后端。
- 离线草稿队列、同步冲突提示、失败重试。
- 隐私模式、金额模糊、错误态、加载态、空态补齐。

验收：

- iOS/macOS 可通过域名访问云端 API。
- 云端 DB 有备份和恢复演练记录。
- 关键业务流程在生产环境可跑通。

## 11. 推荐开发顺序

1. Phase 0：工程骨架。
2. Phase 1：基础账本。
3. Phase 2：信用账户与账单周期。
4. Phase 3：未来现金流。
5. Phase 4：报销系统。
6. Phase 5：分期与订阅。
7. Phase 6：AI 与通知。
8. Phase 7：报表与 CSV 导出。
9. Phase 8：云端部署与稳定化。

实际施工时可以并行：

- 前端 DesignSystem、导航骨架、MoneyText 可与后端 Phase 0-1 并行。
- 报表 UI 骨架可以提前做，但真实数据必须等后端聚合 API。
- AI 页面可先做 `AIPlanCard` 假数据，再接真实 AIAction API。
- 云端部署准备可在 Phase 1 后开始，但正式切域名建议等核心数据模型稳定。

## 12. 第一批应创建的工程任务

1. 建立后端项目骨架和数据库迁移。
2. 建立前端 SwiftUI 跨 iOS/macOS 工程骨架。
3. 定义 OpenAPI/API contract 初版。
4. 落库基础模型：账户、分类、汇率、记录、分类明细、账户变动、审计日志。
5. 写入 USD/CNY = `6.8` 的手动汇率 seed。
6. 完成账户、分类、记账、余额总览 API。
7. 完成 iOS/macOS 的 API Client、DesignSystem v0、基础记账界面。
8. 加入 `CreditStatementCycle` 数据模型设计和迁移草案，避免 Phase 2 返工。

## 13. 暂不做

- 预算模块。
- 自动汇率 API。
- 银行自动同步/抓取。
- 多用户协作账本。
- 投资交易系统。
- V1 的 JSON/SQLite/PDF 导出。
- CloudKit 作为主同步方案。
